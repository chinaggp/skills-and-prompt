# Spec-Master (规范大师) 技能

基于 GitHub Spec Kit 方法论，将"氛围编程"升级为"规范编程"。
核心理念：**先写规范，再写代码。每一行代码都有据可循。**

## 📁 目录约定

Spec-Master 在每个项目中使用以下目录结构：

```
[项目根目录]/
└── .agent/spec/                    # SDD 工作目录
    ├── constitution.md             # 🏛️ 项目宪法（项目特有的基本准则）
    └── [日期+需求名称]/                  # 📦 按需求聚合（每个需求一个目录）
        ├── spec.md                 # 需求规范
        ├── plan.md                 # 技术方案
        └── tasks.md               # 任务清单
```

**模板文件**位于本技能的 `templates/` 目录下：

- `templates/spec.md` — 需求规范模板
- `templates/plan.md` — 技术方案模板
- `templates/tasks.md` — 任务清单模板
- `templates/constitution.md` — 项目宪法模板（首次使用时复制到项目中）

## 🛠️ 核心工作流

### 阶段 0: 初始化（首次在项目中使用时）

1. 检查项目中是否存在 `.agent/spec/constitution.md`
2. 如果不存在：
   - 基于 `templates/constitution.md` 模板创建
   - 根据项目的技术栈和业务场景，**与用户一起定制宪法内容**
   - 宪法需要涵盖：架构准则、编码规范、安全准则、数据库规范、测试要求、性能要求、API 设计规范
3. 如果已存在：直接加载

### 阶段 1: 需求规范 (Specify)

> 🎯 明确 **做什么** 和 **为什么**

1. 与用户沟通需求，理解业务背景和目标
2. 在 `.agent/spec/` 下创建需求目录（简短命名，例如 `积分兑换`）
3. 基于 `templates/spec.md` 模板创建 `spec.md`
4. 填写内容：
   - 背景与动机（为什么要做）
   - 用户故事（给谁用、实现什么能力、带来什么价值）
   - 详细功能描述和业务规则
   - 接口需求概要
   - 验收标准（明确的完成条件）
5. **⏸️ 暂停，提交给用户确认**

### 阶段 2: 技术方案 (Plan)

> 🎯 明确 **怎么做**

1. 基于确认后的 spec.md
2. 使用 `templates/plan.md` 模板创建 `plan.md`
3. 填写技术方案：
   - **首先完成宪法审查清单**（对照 constitution.md 逐项打钩）
   - 数据库表设计（DDL）
   - RESTful 接口设计（路径、方法、请求/响应体）
   - 核心类设计（Entity / VO / DTO / Service / Controller / Mapper）
   - 业务流程描述
   - 依赖与影响分析
   - 风险评估
4. 如有多个可选方案，列出方案对比
5. **⏸️ 暂停，提交给用户确认**

### 阶段 3: 任务拆分 (Tasks)

> 🎯 拆分为 **AI 可逐步执行的原子任务**

1. 使用 `templates/tasks.md` 模板创建 `tasks.md`
2. 按阶段拆分任务：
   - **数据层**: DDL 脚本 → Entity → Mapper → Mapper XML
   - **业务层**: DTO/VO → Service 接口 → ServiceImpl
   - **接口层**: Controller
   - **测试**: 单元测试 → 集成测试
   - **收尾**: 宪法合规检查 → 代码审查
3. 每个任务必须包含：
   - 具体的文件路径
   - 明确的要求和约束
   - 可验证的完成标准
4. **⏸️ 暂停，提交给用户确认**

### 阶段 4: 实现 (Implement)

> 🎯 逐个执行任务，保质保量

1. 按 tasks.md 中的顺序逐个实现
2. 每完成一个任务：
   - 对照宪法检查代码合规性
   - 确保编译通过
   - 更新 tasks.md 中的勾选状态 `- [x]`
3. 所有任务完成后：
   - 运行测试确保全部通过
   - 更新进度追踪表
   - 汇报完成情况

## 🏛️ 关于宪法 (Constitution)

宪法是 Spec-Master 的灵魂。它定义了项目中所有开发工作必须遵循的**不可违背的基本准则**。

**宪法是项目级别的**（不是技能级别的），因为每个项目的技术栈、业务场景和团队规范不同。

**宪法可以演进**：遇到宪法未覆盖的场景时，应与用户讨论是否需要更新宪法。

## 💡 关键原则

1. **宪法至上** — 任何方案和代码都不能违反 constitution.md
2. **逐级确认** — 每个阶段都必须经过用户确认才能进入下一阶段
3. **需求聚合** — 一个需求的所有文档放在同一个目录下
4. **任务原子性** — 每个 Task 应该是独立的、可验证的小工作单元
5. **宪法可演进** — 宪法不是一成不变的，但修改需要慎重讨论

## 💡 使用提示

- 如果需求很小（比如只是加个字段），可以简化流程，跳过 spec 直接写 plan
- 如果需求很复杂，spec 阶段可以拆成多轮对话迭代
- 宪法首次创建后建议提交到 Git，作为项目的"开发宪法"长期维护
