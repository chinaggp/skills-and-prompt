# 阶段模型路由

使用此技能时，应按交付阶段而不是业务领域来选择模型。

保持此技能的边界清晰。它不定义强制性的交付流程，只负责决定当前工作阶段应由哪个模型承担。

## 阶段选择

以下分析阶段工作应选择 `gpt-5.4`：

- 澄清含糊不清的需求
- 阅读代码、配置、日志或文档以建立理解
- 识别约束、风险、回归点或边界情况
- 比较不同方案并给出设计建议
- 在开始实现前将工作拆分为任务
- 判断一个已报告的问题是否真实存在，以及它大概率位于哪里

以下编码阶段工作应选择 `gpt-5.4-mini`：

- 在实现方向已经明确后编辑或生成代码
- 为已达成一致的实现补充或调整测试
- 修复编译、构建、Lint 或相对直接的运行时问题
- 在意图已知的前提下执行定向重构
- 为已经进行中的实现完成验证与收尾工作

## 边界规则

当请求中的核心未知数是“应该改什么”或“应该如何实现”时，将该请求视为分析阶段。

当请求中的核心未知数已经不再是方向问题，而主要任务是安全地执行已选定的变更时，将该请求视为编码阶段。

如果一个请求同时包含分析和实现，应拆分处理：

1. 先使用 `gpt-5.4` 解决不明确的部分。
2. 当实现目标已经具体明确后，切换到 `gpt-5.4-mini`。
3. 只有在出现新的歧义、风险或设计冲突时，才切回 `gpt-5.4`。

## 默认策略

如果当前阶段不明确，默认使用 `gpt-5.4` 进入分析阶段。

只做足以消除不确定性的最小分析。当实现路径已经清晰后，不要继续停留在分析模式。

## 执行规则

只有当模型切换是显式且可验证的，这个技能才真正有意义。

当任务处于分析阶段时，应继续由 `gpt-5.4` 承担工作。

当任务进入实现阶段时，你必须将实现工作委派给一个使用以下配置创建的子代理：

- `spawn_agent`
- `model: "gpt-5.4-mini"`

在某个阶段开始较为实质性的工作前，你必须在回复中显式声明当前阶段和模型。

使用如下固定格式：

- `当前阶段：分析，模型：gpt-5.4`
- `当前阶段：实现，模型：gpt-5.4-mini`

除非实现代理是在显式指定 `model` 字段的情况下创建的，否则不要声称已经完成了模型切换。

如果实现工作暴露出新的歧义、设计冲突或显著风险，应将分析所有权切回 `gpt-5.4`，并在继续之前显式宣布该切换。

如果实现非常小，不需要创建子代理，也不要假装已经发生了切换。在这种情况下，应明确说明任务仍由当前代理处理，并未发生阶段切换。

## 示例

以下情况应先使用 `gpt-5.4`：

- “分析一下这个需求应该怎么改”
- “先读代码，提出方案，确认后再实现”
- “找出这个空指针到底是从哪里来的”

当方向已经确定后，以下情况应使用 `gpt-5.4-mini`：

- “实现已经确认的 API 变更并补上测试”
- “修复这个接口里的空指针问题”
- “完成这个已经确定的重构并做验证”

对于“先分析，再直接实现”这类混合请求，应先由 `gpt-5.4` 开始，概括选定的实现方向后，再继续交由 `gpt-5.4-mini`。

一个可验证的混合流程示例：

1. 先声明 `当前阶段：分析，模型：gpt-5.4`
2. 解决不明确的部分，并总结实现目标
3. 使用 `spawn_agent(... model="gpt-5.4-mini" ...)` 创建实现子代理
4. 再声明 `当前阶段：实现，模型：gpt-5.4-mini`
5. 如果出现新的歧义，则切回 `gpt-5.4`，并显式宣布该切换
